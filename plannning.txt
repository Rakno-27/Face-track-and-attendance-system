========================================
FACETRACK: SYSTEM DOCUMENTATION & KNOWLEDGE TRANSFER
========================================

----------------------------------------
1. PROJECT PURPOSE
----------------------------------------
FaceTrack is an automated, real-time facial recognition student attendance tracking system. 

It was built to eradicate the traditional, time-consuming manual roll-call process in academic environments by allowing a continuous camera feed to seamlessly identify students and securely log their presence during strict class schedules. The system solves problems regarding proxy-attendance (buddy-punching), time-wastage during lecture beginnings, and manual data-entry errors.

----------------------------------------
2. CORE WORKING LOGIC
----------------------------------------
1. **System Initialization:** On boot, the server spins up a background daemon thread that activates the host hardware camera and caches all registered student facial encodings from the SQLite database directly into high-speed memory arrays.
2. **Video Ingestion:** The background thread continuously reads frames from OpenCV. For performance, every 2nd frame is aggressively downscaled and piped to the Deep Learning detector (dlib/face_recognition).
3. **Identification Phase:** Deep learning models extract facial landmarks and compute an affine 128-dimension vector. These vectors are compared mathematically against the cached student encodings using a strict distance tolerance threshold.
4. **Timetable Validation:** When a known face occurs, the system checks the current server time against the hardcoded daily timetable or specific custom override sessions active today.
5. **Data Persistence:** If the time falls within a valid session window (including a 15-minute early buffer), safety checks verify identical records. If clear, a synchronous SQL `INSERT` officially logs the student's attendance.

----------------------------------------
3. SYSTEM ARCHITECTURE
----------------------------------------
*   **Monolithic Backend:** A central Python Flask application (`app.py`) orchestrates endpoints, threading, data access layers, and UI templates.
*   **Hardware Layer (Camera Engine):** A decoupled background `threading.Thread` manages the `cv2.VideoCapture` bindings natively shielding the REST API from I/O starvation.
*   **Authentication Middleware:** A customized `@app.before_request` interceptor natively bounds Session Cookies, CSRF tokens natively, and Role-Based Access Control (RBAC) across Administration subsets. 
*   **Frontend Ecosystem:** A pure HTML/CSS/Vanilla Javascript stack that requests DOM renders, polling the `/api/attendance` endpoints dynamically and natively routing fetch requests via a centralized asynchronous wrapper hooking CSRF tokens globally.

----------------------------------------
4. DATABASE DESIGN
----------------------------------------
The system relies on an optimized SQLite relational matrix orchestrated via Flask-SQLAlchemy (`models.py`).

*   **User:** Manages `username`, `password_hash`, and `role` (Admin/Teacher).
*   **Student:** Manages academic details (`name`, `roll_number`), biological markers (`encoding` as BLOB), and dynamically system-generated sequences (`student_code` formatted precisely as STU-YYYY-XXX). 
*   **Subject:** Simplifies academic class schemas (`name`, `code`).
*   **Session:** Represents timetable components locally. Links via Foreign Key to `Subject`. Contains explicit boundaries (`start_time`, `end_time`, `date`, `is_custom`).
*   **Attendance:** Mapped Intersection Table linking `student_id` (FK) and `session_id` (FK) carrying the exact `timestamp` and `status` string.
